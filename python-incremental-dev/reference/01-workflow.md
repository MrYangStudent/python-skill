# 完整开发工作流

## 增量开发工作流

```
┌─────────────────────────────────────────────────────────────────┐
│                  前置步骤：需求收集与参考索引                       │
├─────────────────────────────────────────────────────────────────┤
│  1. 询问是否存在接口文档 / PRD / 原型                             │
│  2. 询问是否存在可参考的代码流程（相似功能文件路径）                │
│  3. 检查知识库是否存在，比较时间戳判断是否需要重建索引              │
│  4. 确定新增功能的影响范围（涉及哪些模块/文件）                    │
│  5. 确定增量策略优先级（P0内联扩展 → P3新模块）                    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      阶段零：边界确认                            │
├─────────────────────────────────────────────────────────────────┤
│  1. 明确新功能范围                                               │
│  2. 确定影响边界（哪些旧代码可能被触及）                          │
│  3. 确认向后兼容要求                                             │
│  4. 制定"不改动"策略（如必须改动，说明原因）                      │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      阶段一：需求分析                             │
├─────────────────────────────────────────────────────────────────┤
│  - 功能目的与价值                                                 │
│  - 输入→处理→输出                                                 │
│  - 约束条件（性能、安全、兼容性）                                 │
│  - 验收标准                                                       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      阶段二：模块设计                            │
├─────────────────────────────────────────────────────────────────┤
│  1. 根据 P0-P3 策略确定改动方式                                   │
│     P0: 现有文件中内联扩展                                        │
│     P1: 组合现有模块实现                                          │
│     P2: 适配器包装旧代码                                          │
│     P3: 新模块独立目录（末选）                                    │
│  2. Protocol/ABC 设计（面向接口编程，P2-P3 时使用）              │
│  3. 依赖注入（避免硬依赖旧模块）                                   │
│  4. 向后兼容接口                                                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      阶段三：代码实现                             │
├─────────────────────────────────────────────────────────────────┤
│  - Docstring 完整注释                                            │
│  - 异常处理规范                                                   │
│  - 类型注解完整性                                                 │
│  - 并发安全（如涉及）                                             │
│  - 标准库优先封装                                                 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      阶段四：测试生成                             │
├─────────────────────────────────────────────────────────────────┤
│  - 正常路径测试                                                   │
│  - 边界情况测试                                                   │
│  - 错误路径测试                                                   │
│  - Mock 编写（如依赖外部服务）                                    │
│  - 并发安全测试（如涉及并发）                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      阶段五：质量审查                             │
├─────────────────────────────────────────────────────────────────┤
│  只审查新代码文件：                                               │
│  - 异常处理审查                                                   │
│  - 类型安全审查                                                   │
│  - 安全漏洞审查                                                   │
│  - 性能审查                                                       │
│  - 日志规范审查                                                   │
│  - API 设计审查                                                   │
│  - 并发安全审查                                                   │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      阶段六：集成验证                             │
├─────────────────────────────────────────────────────────────────┤
│  1. 类型检查：mypy 或 pyright                                    │
│  2. 静态分析：ruff check                                         │
│  3. 格式化检查：ruff format --check                              │
│  4. 单元测试：pytest --cov                                       │
│  5. 覆盖率：pytest --cov-report=term-missing                    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      阶段七：文档沉淀                             │
├─────────────────────────────────────────────────────────────────┤
│  - 更新模块 README                                                │
│  - 添加 Docstring 注释                                           │
│  - 记录设计决策                                                  │
│  - 更新 CHANGELOG                                                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                阶段八：知识沉淀（新增）                            │
├─────────────────────────────────────────────────────────────────┤
│  1. 识别本次新增的可复用模式（工具函数、设计模式等）                │
│  2. 使用 python knowledge.py add 命令保存条目                     │
│  3. 供下次开发时 search/check-stale 复用                          │
└─────────────────────────────────────────────────────────────────┘
```

## 新旧代码隔离检查清单

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 新功能是否有独立目录/模块 | □ | 便于单独测试和审查 |
| 是否通过 Protocol/ABC 与旧代码交互 | □ | 降低耦合 |
| 是否需要修改已有接口 | □ | 如需修改，列出所有影响点 |
| 新功能是否有向后兼容要求 | □ | API/数据格式兼容 |
| 是否需要版本迁移逻辑 | □ | 数据库、配置等迁移 |

## 快速启动命令

```bash
# 0. 初始化知识库（首次使用）
python tools/knowledge/knowledge.py init

# 1. 检查知识库状态
ls -la project-knowledge/INDEX.md 2>nul || echo "知识库不存在，将首次创建"

# 1. 搜索知识库（检索可复用代码）
python tools/knowledge/knowledge.py search <功能关键词>

# 2. 检查知识条目是否过期
python tools/knowledge/knowledge.py check-stale .

# 3. 知识库完整性检查
python tools/knowledge/knowledge.py lint

# 4. 创建新模块目录
mkdir -p features/<feature_name>

# 5. 新功能代码结构示例
features/
├── myfeature/
│   ├── __init__.py          # 模块入口
│   ├── myfeature.py         # 主逻辑
│   ├── test_myfeature.py    # 测试
│   └── adapter.py           # 适配器（如需要）

# 6. 运行测试
pytest --cov features/myfeature

# 7. 静态检查
ruff check features/myfeature
ruff format --check features/myfeature

# 8. 保存知识条目
python tools/knowledge/knowledge.py add -t "[标签] 功能名 - 用途" -s "features/myfeature/" -p "用途描述"
```

## 问题定义要素表

在开始编码之前，必须澄清以下要素：

| 要素 | 问题 | 示例 |
|------|------|------|
| 目的 | 解决谁的什么问题？ | 解决运维人员批量管理服务器的问题 |
| 场景 | 在什么情况下触发？ | 在凌晨批量执行部署任务时 |
| 结构 | 输入→处理→输出？ | 服务器列表→并行执行→结果汇总 |
| 约束 | 性能、安全、技术边界？ | 最多 100 台，支持 SSH key 认证 |
| 验收 | 怎么算"完成"？ | 所有服务器部署成功，返回详细日志 |

**如果需求不清晰，优先向用户提出不超过 3 个澄清问题。**

## 模块拆分原则

将大功能拆为独立的、可验证的微模块：

```
Feature/
├── module_1/
│   ├── __init__.py
│   ├── module_1.py       # 业务逻辑
│   └── test_module_1.py  # 单元测试
├── module_2/
│   ├── __init__.py
│   ├── module_2.py
│   └── test_module_2.py
└── test_integration.py   # 集成测试
```

**每个模块原则**：
- 单一职责（SRP）
- 有明确的输入输出
- 可独立验证
- 最小依赖

## 代码生成循环

循环执行以下步骤：

```
┌─────────────────────────┐
│  1. 设计思路 + 数据结构  │
│     （等用户确认）        │
├─────────────────────────┤
│  2. 生成完整代码         │
│     （含异常处理）        │
├─────────────────────────┤
│  3. 生成单元测试          │
│     （含边界测试）        │
├─────────────────────────┤
│  4. 验证测试通过          │
│     （pytest）            │
└─────────────────────────┘
         ↓
    下一个模块
```

## 集成验证命令

```bash
# 1. 静态检查
ruff check .
ruff format --check .

# 2. 单元测试 + 覆盖率
pytest --cov --cov-report=term-missing

# 3. 集成测试（如有）
pytest -m integration

# 4. 类型检查（如配置了 mypy/pyright）
mypy .

# 5. 知识沉淀：保存新增的可复用函数/模式
python tools/knowledge/knowledge.py add -t "[模式] 本次新增模式名" -s "pkg/xxx/xxx.py" -p "用途描述"
python tools/knowledge/knowledge.py list  # 确认条目已更新
```
